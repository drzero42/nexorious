# Frontend Port — Design Spec

## Overview

The React SPA lives in the Python (nexorious) repository. This spec covers porting it into the Go repository (`nexorious-go`) as the definitive frontend, wired against the Go backend. The frontend is kept as-is — no rewrite. The work is: copy the source tree, adjust the project layout to fit the Go repo's `ui/` structure, and apply the small set of changes required by the go-port design spec.

## Build Integration

### Makefile

```makefile
frontend:
	cd ui/frontend && npm install && npm run build
```

`npm run build` runs `vite build && tsc --noEmit`. Vite reads `vite.config.ts` (in `ui/frontend/`) and outputs the built SPA to `ui/frontend/dist/`.

### `ui/ui.go`

`ui/ui.go` already declares four embed vars (`UIBox`, `MigrateBox`, `DBErrorBox`, `SetupBox`). Only `UIBox` changes — its embed path updates from `dist` to `frontend/dist`:

```go
//go:embed all:frontend/dist
var UIBox embed.FS
```

The Go serving code (`spaHandler` in `internal/api/router.go`) must be updated from `fs.Sub(ui.UIBox, "dist")` to `fs.Sub(ui.UIBox, "frontend/dist")`. The other three embed vars are unchanged.

This single change covers all static assets served through the SPA handler — including the logo files. Logos are not served via a dedicated backend route; Vite copies `ui/frontend/public/logos/` into `ui/frontend/dist/logos/` at build time, and the SPA handler serves them at `/logos/...` from the embedded FS.

### `package.json`

Update the `name` field from `"frontend"` to `"nexorious-ui"`. No functional impact.

### devenv.nix

No changes needed — Node 24 is already provided.

## Logo URL Migration

The Python backend stored full URL paths in the `icon_url` DB column (e.g., `/static/logos/platforms/pc-windows/pc-windows-icon-light.svg`) and served the files at `/static/logos/`. The Go backend stores only the bare filename in the `icon` column (e.g., `pc-windows-icon-light.svg`).

With logos bundled into the SPA dist, they are served at `/logos/...` (no `/static/` prefix). The platforms and storefronts API responses must be updated to return a constructed `icon_url` field so the frontend receives a ready-to-use URL.

### Backend: construct `icon_url` in API responses

The `Platform` and `Storefront` models currently expose `Icon *string` with `json:"icon"`. The API responses need to include `icon_url` as a full path instead of the raw filename. Two options:

- Add an `IconURL() *string` method on the model that constructs the path, and expose it via a response DTO — **preferred**, keeps the model clean
- Change `json:"icon"` to `json:"icon_url"` and have the handler rewrite the value before returning

The directory segment in the path is the platform/storefront's `Name` field, which is the TEXT primary key slug (e.g. `"pc-windows"`, `"steam"`). The constructed paths follow the same directory structure as the logo files:
- Platforms: `/logos/platforms/<platform.Name>/<icon-filename>`
- Storefronts: `/logos/storefronts/<storefront.Name>/<icon-filename>`

When `icon` is NULL, `icon_url` should be `null` in the response.

### Frontend: update hardcoded logo URL references

The Python frontend has hardcoded `/static/logos/` paths in several places. These must be updated to `/logos/`:

- `src/types/sync.ts` — `iconUrl` constants for Steam, Epic, GOG, PlayStation Store
- `src/routes/_authenticated/admin/platforms.tsx` — placeholder text on icon URL inputs
- Test fixtures in `src/components/games/game-card.test.tsx` and `src/components/games/game-list.test.tsx` — `icon_url` values in mock data

## IGDB Availability

The Go backend makes IGDB credentials optional. When unconfigured, `GET /health` returns `igdb_configured: false`, IGDB-dependent endpoints return `503 Service Unavailable`, and scheduled metadata refresh jobs do not run. The frontend must surface this clearly without blocking non-IGDB usage.

### Data fetching

A `useHealthStatus` TanStack Query hook fetches `GET /health`. It is configured with a 60-second stale time and refetches on window focus. Components that need IGDB availability call this hook and receive the cached result — no prop drilling or context required.

```ts
// src/hooks/use-health-status.ts
export function useHealthStatus() {
  return useQuery({
    queryKey: ['health'],
    queryFn: () => api.get('/health'),  // returns { igdb_configured: boolean, ... }
    staleTime: 60_000,
    refetchOnWindowFocus: true,
  })
}
```

### Global banner

A persistent, non-dismissible banner is rendered in `_authenticated.tsx` (the authenticated layout route) when `igdb_configured === false`. It sits below the nav, above the page content.

Banner text: *"IGDB is not configured — game search and scheduled metadata refresh are unavailable. An administrator needs to set `IGDB_CLIENT_ID` and `IGDB_CLIENT_SECRET`."*

The banner is persistent (not dismissible) — it stays visible as long as IGDB is unconfigured so users do not forget why features are missing.

### Inline disabled states

Two specific interaction points are disabled when `igdb_configured === false`:

- **Add game flow** (`src/routes/_authenticated/games/add.*`) — the IGDB search tab or button is rendered in a disabled state with a tooltip: *"IGDB not configured"*
- **Any other IGDB import trigger** — same treatment

Non-IGDB features (browsing the collection, editing user games, managing tags/platforms, import/export, sync) are unaffected regardless of IGDB state.

### 503 handling

IGDB endpoints return `503` when IGDB credentials are not configured. At call sites for IGDB-adjacent queries (game search, IGDB import), a `503` response is caught and surfaced as a clear user-facing message ("IGDB is currently unavailable") via the existing toast/notification system, rather than a generic error.
