# SLSA Build L2 + SBOM for the release pipeline

**Date:** 2026-06-15
**Status:** Design — pending implementation plan
**Scope:** `.github/workflows/release-artifacts.yaml`, a new root `SECURITY.md`, and a GitHub repo setting (private vulnerability reporting).

## Goal

Raise the supply-chain integrity of every artifact published by a `release`
event from effectively **SLSA Build L0** to **SLSA Build L2**, add **SBOMs** for
the runtime image and the frontend dependency tree, and make the result both
**self-verifiable in CI** and **user-verifiable** via `SECURITY.md`.

"Build L2" here means: provenance is generated by a hosted build platform
(GitHub Actions) and **signed** with the workflow's OIDC identity via Sigstore,
attached to each published artifact, and resolvable/verifiable by consumers with
`gh attestation verify`. We are explicitly **not** pursuing Build L3 (isolated
reusable-workflow builder) — the marginal benefit does not justify restructuring
the pipeline for a self-hosted, single-maintainer application.

### Non-goals

- No SLSA Build L3 (`slsa-github-generator` reusable workflows).
- No change to the artifacts themselves — only added provenance / SBOM / signing.
- No per-binary or per-package SBOMs, and no chart SBOM (see Decisions).
- No new behavior on PR builds beyond optional unsigned SBOM *generation*; all
  signing/attestation/push remains release-only.

## Threat model addressed

Tampering with a published artifact after it leaves the build:

- The `ghcr.io/drzero42/nexorious` image and its `:latest` tag.
- Release binaries / `.deb` / `.rpm` attached to the GitHub Release.
- The Helm chart at `oci://ghcr.io/drzero42/charts`.

Signed provenance lets a consumer confirm an artifact was produced by *this*
repo's workflow from a specific commit, not swapped or rebuilt elsewhere. SBOMs
let a consumer enumerate the dependency surface for vulnerability scanning.

## Decisions (settled during brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Target level | Build **L2**, not L3 | L2 is cheap, keyless, high-value; L3 needs a pipeline rewrite for marginal gain here. |
| Provenance coverage | **Every** published artifact (binaries, both packages, image, chart) | Per-artifact and nearly free. |
| SBOM coverage | **Image + source**, no per-binary, no chart | Image SBOM already contains the full Go module tree + Alpine OS + python. Per-binary SBOM is a strict subset (redundant). Source SBOM is the **only** way to capture the npm/frontend tree, which is compiled into the binary and invisible to a binary/image scan. |
| Publish format | **Signed Sigstore attestations**, not plain files | Plain SBOM files are not L2 — nothing cryptographically ties them to the build. |
| SBOM format | **SPDX JSON** | Aligns with GitHub's tooling and `attest-sbom`. |
| Verification surface | **CI self-verify job** + **`SECURITY.md`** | Catch a broken pipeline before users; give users copy-pasteable verify commands. (Admin-guide section declined.) |
| Disclosure channel | **GitHub private vulnerability reporting** | No email to maintain; integrates with advisories. Requires enabling the repo setting. |
| Trigger scope | Attestation/SBOM/publish steps are **release-only** | OIDC publish path and ghcr push only make sense on a real release. |

## Architecture — changes per existing job

All new attestation/SBOM/publish steps are gated `if: github.event_name == 'release'`.
PR builds keep building + smoke-testing unchanged.

### `build` job (binaries + packages)

Add job permissions `id-token: write` and `attestations: write` (keeps
`contents: read`).

1. **Source SBOM** — generate with `anchore/sbom-action` over `dir:.`, SPDX JSON,
   written to `dist/nexorious_<ver>_sbom.source.spdx.json`. Because it lands in
   `dist/` and matches the `nexorious*` glob, it is automatically included in
   `sha256sums.txt` and uploaded as a release asset by the existing `upload` job.
2. **Provenance** — `actions/attest-build-provenance` with
   `subject-path: 'dist/nexorious*'`. One signed provenance bundle whose subjects
   are every binary, both packages, and the source SBOM file.

Ordering: build binaries → build packages → generate source SBOM → checksums →
attest → upload artifacts.

### `image` job

Add `id-token: write` + `attestations: write` (already has `packages: write`).

1. Capture the manifest-list digest from `docker/build-push-action`'s `digest`
   output.
2. **Provenance** — `actions/attest-build-provenance` with `subject-name`
   (`ghcr.io/drzero42/nexorious`), `subject-digest: <digest>`,
   `push-to-registry: true`.
3. **Image SBOM** — generate with syft over the pushed image (SPDX JSON), attach
   via `actions/attest-sbom` (`subject-digest` + `sbom-path`,
   `push-to-registry: true`). Captures Go modules + Alpine OS packages + python.
4. Leave BuildKit's own `provenance:` / `sbom:` outputs **off** so there are not
   two competing attestation mechanisms on the multi-arch manifest.

Attesting the manifest-list (index) digest is the standard L2 approach and is
what `gh attestation verify oci://…:<tag>` resolves to.

### `chart` job

Add `id-token: write` + `attestations: write` (already has `packages: write`).

1. Capture the `Digest:` line from `helm push` output.
2. **Provenance** — `actions/attest-build-provenance` with
   `subject-name: ghcr.io/drzero42/charts/nexorious`, the captured digest,
   `push-to-registry: true`.

No chart SBOM.

### New `verify-attestations` job

Release-only, `needs: [image, upload, chart]`. Uses `gh attestation verify`
(`--repo drzero42/nexorious`) against:

- the pushed image by tag (`oci://ghcr.io/drzero42/nexorious:<ver>`),
- one downloaded release binary asset,
- the pushed chart (`oci://ghcr.io/drzero42/charts/nexorious:<ver>`).

Fails the run if any attestation is missing or unresolvable. Needs only repo read
+ `GITHUB_TOKEN`; no OIDC.

## `SECURITY.md` (new, repo root)

Two sections:

1. **Reporting a vulnerability** — directs reporters to GitHub's private
   vulnerability reporting ("Report a vulnerability" on the Security tab). This
   requires enabling *Private vulnerability reporting* in repo settings (a
   one-time toggle, called out as an implementation step).
2. **Verifying releases** — copy-pasteable commands:
   - `gh attestation verify oci://ghcr.io/drzero42/nexorious:<ver> --repo drzero42/nexorious`
   - `gh attestation verify <downloaded-binary> --repo drzero42/nexorious`
   - `gh attestation verify oci://ghcr.io/drzero42/charts/nexorious:<ver> --repo drzero42/nexorious`
   - downloading the image SBOM and the source SBOM release asset.

## Action versions

Pins are **not** taken from memory. Before implementation, verify the current
latest major of each new action against its source/marketplace and pin to that
major (matching the repo's existing `@vN` house style, not SHA pins):

- `actions/attest-build-provenance`
- `actions/attest-sbom`
- `anchore/sbom-action`

## Testing & validation

This is CI configuration, so there are no unit tests. Validation layers:

- The `verify-attestations` job is the executable check that attestations exist
  and resolve.
- As much as possible is exercised on the PR path without signing — notably
  **SBOM generation** can run on PRs (unsigned) to catch syft/config breakage
  before a release; only the `attest-*` and push steps stay release-gated.
- The genuine end-to-end proof is the first tagged release after merge.

**Known blind spot:** the OIDC signing + ghcr publish path cannot run on PRs, so
that portion is first exercised on a real release. Mitigation: keep the
release-only surface minimal and mechanically simple, and verify it immediately
via the `verify-attestations` job on that first release.

## Risks

- **First-release-only signing path** — see blind spot above.
- **`helm push` digest parsing** — depends on the `Digest:` line in helm output;
  guard with a non-empty check so a format change fails loudly rather than
  attesting an empty subject.
- **Multi-arch attestation scope** — we attest the index digest, not per-arch
  manifests; acceptable and standard for L2.
- **Private vulnerability reporting toggle** — `SECURITY.md` references a feature
  that must be enabled in repo settings; if left disabled the "Report a
  vulnerability" button is absent. Listed as an explicit implementation step.
